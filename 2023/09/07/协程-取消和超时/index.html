协程-取消和超时

取消协程的执行

fun main() = runBlocking {} 使用 runBlocking 函数启动一个新的协程。runBlocking 函数将阻塞当前线程，直到所有启动在该协程中的协程都执行结束。

val job = launch {} 使用 launch 函数启动一个新的协程。launch 函数将返回一个 Job 对象，该对象表示该协程

job.cancel() 取消 job 协程。

job.join() 等待 job 协程执行结束。

在 main() 函数中，首先使用 launch 函数启动了一个新的协程。该协程将执行一个循环，在循环中每隔 500 毫秒打印一条信息。

然后，main() 函数延迟 1300 毫秒。这意味着 main() 函数将在 1300 毫秒后继续执行。

在 1300 毫秒后，main() 函数打印一条信息，表示它已经厌倦了等待。然后，main() 函数取消该协程。

最后，main() 函数等待该协程执行结束。

Finally总在协程结束执行，无论是结束还是取消

cancel cancelAndJoin

cancel 和 cancelAndJoin 是 Kotlin 协程中用于取消协程的两个不同方法，它们有一些区别和适用场景：

cancel

  • 作用：cancel 用于发送取消请求给协程。它会将协程的状态标记为被取消，但并不等待协程实际结束。协程在取消请求后会继续执行，除非在协程体内检查了取消状态，并在需要时自行处理取消。
  • 使用场景：
    • 当你希望取消一个协程，但不需要等待其结束时，可以使用 cancel。
    • 适用于不需要立即等待协程执行完毕的场景，比如后台任务。

cancelAndJoin

  • 作用：cancelAndJoin 同样发送取消请求给协程，但它会等待协程执行结束。它是 cancel 和 join 操作的组合，即取消并等待。
  • 使用场景：
    • 当你希望取消一个协程，并等待其执行完毕时，可以使用 cancelAndJoin。
    • 适用于需要确保协程被取消并且等待其完成的场景，比如在主协程中等待子协程的结束。

  3. 由于 withTimeout 设置的时间是 1300 毫秒，而每次循环挂起 500 毫秒，因此总共循环 2 次。之后，超时时间已过，withTimeout 将抛出 TimeoutCancellationException。

注意：在实际的应用中，使用超时限制可能会对正在执行的任务产生影响，因此在使用时需要谨慎考虑。

  3. 劣势：如果在超时时不想抛出异常，而是希望得到一个结果，这样的需求不适合 withTimeout。

第二个代码片段：

  1. 行为：withTimeoutOrNull 在指定的时间内执行代码块，如果超时，会返回 null，整个 main 协程不会被取消。
  2. 优势：适用于希望在超时时获得一个可空的结果而不是抛出异常的场景，使得代码更加灵活，可以根据返回结果进行判断。
  3. 劣势：如果强烈要求在超时时抛出异常，或者不希望执行超时后的代码，那么 withTimeoutOrNull 可能不太适用。

选择建议：

  • 如果希望在超时时抛出异常，而不关心返回结果，可以使用 withTimeout。
  • 如果希望在超时时获得一个可空的结果，并根据结果判断下一步操作，可以使用 withTimeoutOrNull。

    • 代码首先定义了一个简单的 Resource 类，用于模拟可关闭资源。
    • 使用 withTimeout 启动了 10,000 个协程，每个协程在其 withTimeout 块内创建 Resource。
    • withTimeout 设置了 60 毫秒的超时，而 delay(50) 造成了每个协程内的代码块执行时间较长。
  2. 问题示例：
    • 由于异步超时的影响，有时 withTimeout 的超时事件可能发生在块已经完成后。
    • 这就导致了资源泄漏，因为资源在块内被获取，但由于异步超时，没有机会在块外释放。
  3. 解决方案：
    • 修改代码，将 Resource 的引用存储在变量中而不是直接从 withTimeout 返回。
    • 在 finally 块内检查引用是否为空，如果非空则释放资源。这样就确保了在任何情况下都会释放资源，避免了异步超时导致的泄漏。
  4. 最终输出：
    • 修改后的代码总是输出零，这表示资源没有泄漏。这是通过确保在超时时不返回资源，而是存储在变量中，然后在 finally 块内检查变量的非空性，以释放资源而实现的。

总体来说，这个例子突出了在异步环境中使用 withTimeout 时可能出现的问题，并提供了一个解决方案，以确保资源在任何情况下都能得到正确释放。
