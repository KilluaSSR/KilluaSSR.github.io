协程上下文与调度器

所有的协程构建器诸如 launch 和 async 接收一个可选的 CoroutineContext 参数，它可以被用来显式的为一个新协程或其它上下文元素指定一个调度器。

非受限调度器 vs 受限调度器

Dispatchers.Unconfined 非受限的调度器非常适用于执行不消耗 CPU 时间的任务，以及不更新局限于特定线程的任何共享数据（如UI）的协程。

在上述代码中，launch(Dispatchers.Unconfined) 使用了 Dispatchers.Unconfined 调度器。

  1. launch(Dispatchers.Unconfined) { ... }: 启动了一个非受限的协程，它将在调用它的线程中运行。
    • println("Unconfined : I'm working in thread ${Thread.currentThread().name}"): 输出一条消息，指示协程正在主线程中运行。
    • delay(500): 添加了一个延迟，但由于使用的是非受限的调度器，这个延迟并不会导致协程挂起，因此它仍然在主线程中执行。
    • println("Unconfined : After delay in thread ${Thread.currentThread().name}"): 输出一条消息，指示延迟后协程切换到其他线程运行。
  2. launch { ... }: 启动了另一个协程，这次没有指定调度器，因此它将继承其父协程的上下文，即 runBlocking。
    • println("main runBlocking: I'm working in thread ${Thread.currentThread().name}"): 输出一条消息，指示协程正在主线程中运行。
    • delay(1000): 添加了一个更长的延迟。
    • println("main runBlocking: After delay in thread ${Thread.currentThread().name}"): 输出一条消息，指示延迟后协程仍然在主线程中运行。

总体而言，Dispatchers.Unconfined 允许协程在调用线程中执行，而不受限于特定的线程池。这在某些情况下可能很有用，但需要注意，如果在协程中执行的任务很耗时或涉及阻塞 I/O 操作，可能会影响到整体性能。

Dispatchers.Unconfined 是 Kotlin 协程中的一个调度器，它的作用是让协程在调用它的线程中运行，而不受限制于特定的线程池。具体来说，它的行为有以下特点：

  1. 在线程中调用的线程中运行： 协程使用 Dispatchers.Unconfined 调度器时，它将在调用该协程的线程上执行。这使得协程可以直接使用调用线程的上下文，而不受限于特定的线程池。
  2. 不受线程限制： 与其他调度器（如 Dispatchers.Default 或 Dispatchers.IO）不同，Dispatchers.Unconfined 不会将协程限制在特定的线程池中。因此，它对于执行不消耗太多 CPU 时间、不涉及阻塞 I/O 操作的轻量级操作很有用。

需要注意的是，Dispatchers.Unconfined 的使用应谨慎，因为它可能导致一些意外的结果，特别是在多线程环境下。如果协程执行涉及到阻塞 I/O 操作或长时间的 CPU 计算，使用其他调度器可能更为合适，以充分利用线程池的优势。
